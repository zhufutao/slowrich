# 慢富(SlowRich) 技术架构评审计划

> 评审人：代码评审官 | 日期：2026-05-17 | 状态：待确认

## 评审范围

基于产品经理产出的 PRD.md、api-spec.md、data-model.md，对以下6个维度进行架构评审：

### 1. Cloudflare 部署方案评审

**评审要点：**
- Workers + D1 + Pages 三层架构是否合理，各层职责边界是否清晰
- Workers 单请求执行时间限制（CPU 10ms免费/30ms付费）对回测计算的影响——PRD要求回测<30s，但Workers有执行时长上限
- D1 10GB 单库限制与行情数据增长预估的匹配度（data-model估算500只×10年≈500MB，但未考虑增长缓冲）
- Workers Cron Trigger 的可靠性评估——市场温度抓取任务是否适合Cron，失败后的恢复机制
- Pages 静态资源部署 + Workers API 的前后端分离方案的网络延迟
- 冷启动对API响应时间的影响（PRD要求<500ms）

**潜在风险：**
- ⚠️ **回测计算是最大风险点**：Workers CPU时间限制可能导致回测无法在单次请求中完成，需要考虑Durable Objects / Queue / 分片计算等方案
- ⚠️ D1写入并发限制可能影响行情数据批量下载性能

### 2. 数据库设计评审

**评审要点：**
- 表结构设计是否满足D1兼容性约束（无RIGHT JOIN/FULL OUTER JOIN，无存储过程/触发器）
- 索引设计是否覆盖核心查询路径
- ID生成策略（u_xxx/s_xxx等前缀+随机字符串）的唯一性保证和碰撞风险
- JSON字段（params、recommendations）的查询灵活性与性能权衡
- daily_quotes 表的增长预估和分表/归档策略是否充分
- backtest_annual_stats 的复合唯一索引 `(backtest_id, year, strategy_key)` 是否合理
- ON DELETE CASCADE 的使用是否合理（daily_quotes 随 stocks 删除、annual_stats 随 backtest_results 删除）
- 缺失的索引：download_tasks 的 user_id、backtest_results 的 created_at

**潜在风险：**
- ⚠️ daily_quotes 缺少 (stock_id, trade_date) 范围查询优化索引说明
- ⚠️ backtest_annual_stats 单次回测可能产生大量行（15跌幅×12卖出×10年=1800行），多次回测后表膨胀
- ⚠️ 未考虑回测中间交易明细的存储（资金曲线需要逐笔交易记录，当前设计只有年度汇总）

### 3. API 设计评审

**评审要点：**
- RESTful 规范一致性（资源命名、HTTP方法语义）
- 认证方案：JWT Bearer Token，7天有效期，无刷新Token机制
- 分页规范统一性
- 错误码体系完整性
- 回测接口的异步设计：POST创建 → GET轮询结果，是否需要WebSocket/SSE推送
- 断点续传接口 resume 和 retry 的语义区分是否清晰
- 批量操作支持（如批量添加股票、批量下载）
- API 版本控制策略缺失

**潜在风险：**
- ⚠️ **JWT无刷新机制**：7天过期后强制重新登录，用户体验差，应增加Refresh Token
- ⚠️ **回测结果轮询效率**：前端需不断轮询GET /backtest/:id，建议增加完成通知机制
- ⚠️ 缺少速率限制（Rate Limiting）的API层设计
- ⚠️ 行情查询接口按stock_id路径参数，缺少批量查询能力

### 4. 数据源方案评审

**评审要点：**
- AKShare（主）+ Tushare（备）的主备切换逻辑是否健壮
- 数据格式统一映射层的设计（两个数据源字段名差异的适配）
- AKShare 无需Token但接口稳定性如何？频率限制？
- Tushare 需要Token，Token管理策略（存储、刷新、多用户共享）
- 下载任务的断点续传实现——如何检测D1中已有数据的最后日期
- 网络超时和重试策略的配置化
- 数据源健康检查机制

**潜在风险：**
- ⚠️ AKShare 接口变更频繁（Python包更新快），需考虑版本锁定和兼容层
- ⚠️ Tushare 有积分/权限限制，大量下载可能被限流
- ⚠️ 主备切换时数据一致性（两个数据源同一日数据可能有微小差异）

### 5. 回测算法架构评审

**评审要点：**
- 回测算法的正确性：跌幅触发条件（收盘跌幅≥阈值时以收盘价买入）是否合理
- 卖出策略B"盘中浮盈卖出"用最高价模拟盘中价的合理性——实际无法保证在最高价卖出
- 参数组合的笛卡尔积规模：15个跌幅阈值 × (1个次日收盘 + 12个浮盈档位) = 195个策略组合，每个需遍历10年×240天数据
- 计算复杂度估算：195策略 × 2400交易日 ≈ 468,000次计算，在Workers CPU限制内是否可行
- 综合评分公式：年化收益率×40% + 胜率×30% + (1-最大回撤)×30% 的合理性
- 年度资金连续性问题：后一年的年初资金=前一年年末资金，跨年度策略一致性
- 最大回撤计算方式需明确（按资金曲线还是按单笔交易）

**潜在风险：**
- ⚠️ **Workers无法在单次请求中完成回测计算**，这是架构级风险，需要考虑：
  - 方案A：Durable Objects 长时间运行
  - 方案B：Queue 消息队列分片计算
  - 方案C：回测计算拆分为多个子任务
- ⚠️ 用最高价模拟浮盈卖出存在前视偏差（Look-ahead Bias），应在评审报告中指出
- ⚠️ 缺少回测中间交易记录的持久化设计（资金曲线需要）

### 6. 安全性评审

**评审要点：**
- 认证安全：密码bcrypt哈希、JWT签名算法、Token存储（localStorage的XSS风险）
- 权限控制：角色权限（admin/user）的中间件校验，API层是否统一鉴权
- 默认管理员账号 admin@666/666666 的安全性——弱密码风险
- 输入校验：SQL注入防护（D1参数化查询）、XSS防护、CSRF防护
- 敏感数据：密码不可明文返回、Token不可泄露
- 集思录爬虫的合规性——robots.txt、服务条款、频率限制
- CORS 配置策略
- HTTPS 强制

**潜在风险：**
- ⚠️ **默认管理员弱密码**：666666 极易被暴力破解，建议首次登录强制修改
- ⚠️ **JWT存储在localStorage**：存在XSS攻击窃取Token风险，建议HttpOnly Cookie
- ⚠️ **集思录爬虫合规风险**：需确认robots.txt和服务条款，避免法律风险
- ⚠️ 缺少登录失败次数限制（暴力破解防护）
- ⚠️ 缺少CORS白名单配置

## 评审方法

1. 文档交叉审查：PRD ↔ API Spec ↔ Data Model 三份文档一致性校验
2. D1限制约束验证：所有设计是否满足D1已知限制
3. Workers限制约束验证：计算密集型任务（回测）是否超出Workers能力边界
4. 安全基线检查：OWASP Top 10 对照审查
5. 性能可行性评估：关键路径的延迟和吞吐量估算
